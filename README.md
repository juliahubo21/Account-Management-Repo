# Account Management Repo

Daily scanner that surfaces upcoming meetings and email exchanges between Motive Partners and tracked accounts in Affinity, posting new items to Slack `#account-management` once per day at 10:00 America/New_York.

## How it works

1. GitHub Actions cron fires twice daily (14:00 UTC and 15:00 UTC) and runs only when the local New York hour is `10`. This handles DST without configuration changes.
2. The job installs `uv` and the Claude Code CLI, renders `.mcp.json` from `.mcp.json.tmpl` with secrets, and runs `claude --print` against `prompts/daily-scan.md`.
3. The prompt walks the Affinity list **Account Management Master**, pulls meetings and notes for each account in the next 21 days, dedups against the last ~30 days of `#account-management` history (via embedded `<!-- key: ... -->` markers), and posts each new item.
4. If nothing new is found, the job posts nothing — silence is the success signal.

## Required GitHub repo secrets

Set these in **Settings → Secrets and variables → Actions → New repository secret**:

| Secret | Required | Purpose |
| --- | --- | --- |
| `AFFINITY_API_KEY` | yes | Affinity API key (Settings → API in Affinity) |
| `SLACK_BOT_TOKEN` | yes | Slack bot user OAuth token (`xoxb-…`) for "Julia's App", invited to `#account-management`. Required scopes: `channels:history`, `channels:read`, `chat:write`. |
| `ANTHROPIC_API_KEY` | one of the two | Anthropic API key for headless Claude Code runs. |
| `CLAUDE_CODE_OAUTH_TOKEN` | one of the two | Alternative to `ANTHROPIC_API_KEY` — generated by `claude setup-token` locally. Only set one of these two. |

`SLACK_TEAM_ID` is **not** a required secret — the workflow derives it at runtime from `auth.test`.

## First-run checklist

1. Add the four secrets above.
2. Ensure the Slack bot is a member of `#account-management`.
3. Open **Actions** in the GitHub UI and run **Account Management Daily Scan** via **Run workflow** (workflow_dispatch). The schedule gate is bypassed for manual runs, so you can test any time.
4. Watch the run logs. The first successful run should either post one message per new interaction in the next 21 days, or post nothing if there are no in-window Motive ↔ account interactions.
5. After the test passes, the daily 10:00 ET run is automatic.

## Files

- `.github/workflows/account-mgmt-scan.yml` — the scheduled + manual workflow
- `.mcp.json.tmpl` — MCP server config template (rendered with secrets at runtime; `.mcp.json` itself is gitignored)
- `prompts/daily-scan.md` — the per-run instruction the agent executes
- `.gitignore` — keeps the rendered `.mcp.json` out of git

## Editing the prompt

The behavior of each run is determined entirely by `prompts/daily-scan.md`. To change what the scanner does (window length, which list, message format, dedup strategy), edit that file and commit. No workflow changes needed.

## Known caveats

- **Email exposure via the Affinity MCP.** The MCP exposes meetings and notes as first-class objects, but does not expose email chains directly as a separate object. The prompt treats Affinity notes as the proxy for logged email exchanges. If a critical email category is missing in practice, swap to a direct Affinity REST API call inside the prompt or report it back to Affinity via the MCP's `send_feedback` tool.
- **Affinity account URL construction.** The prompt uses any URL field returned by `get_company_info` if available, otherwise falls back to plain account name. If you want consistent hyperlinks, hard-code your workspace subdomain into the prompt template (`https://<workspace>.affinity.co/companies/<id>`).
- **DST.** Handled by running at both 14:00 and 15:00 UTC and gating on America/New_York hour `== 10`. No action needed at DST transitions.
- **Dedup window.** The scanner reads the last ~200 messages of `#account-management`. If your channel volume is high, increase the limit in `prompts/daily-scan.md` step 3.
